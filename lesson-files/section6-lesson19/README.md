# Section 6 수업 19 – 내 비서를 지키는 최소한의 방어 전략

이 수업은 이미 잘 동작하고 있는 OpenClaw 비서를 **“조금 더 안전하게 만드는 작업”**에 초점이 있습니다.  
기능을 늘리는 대신, API 키와 토큰 같은 민감 정보가 최대한 평문으로 노출되지 않도록 설정을 바꿉니다.

---

## 이 수업의 목표

1. `auth-profiles.json`에서 **Gemini API 키를 제거**한다.
2. Gemini API 키를 **호스트(우분투)의 환경변수에서 읽어와 컨테이너로 전달**하는 방식으로 변경하도록 가이드한다.
3. `docker-compose.yml`의 `command:` 설정에서 `--allow-configured` 옵션을 제거해,  
   불필요하게 넓은 권한을 줄이는 방향으로 조정한다.
4. **토큰 비밀번호도 환경변수로 전달**되도록 가이드해서, 평문 노출을 최소화한다.

---

## 변경 전 상태 요약

수업을 시작하기 전에는 보통 다음과 같은 상태를 가정합니다.

- `auth-profiles.json` 안에 Gemini API 키가 **직접 문자열로 기록**되어 있다.
- `docker-compose.yml`의 `environment:`에 민감 정보가 직접 하드코딩되어 있거나,  
  혹은 `auth-profiles.json`에 의존하는 형태로 동작한다.
- `docker-compose.yml`의 `command:` 아래에 `--allow-configured` 옵션이 포함되어 있어,  
  필요 이상으로 설정 변경을 허용할 여지가 있다.
- 토큰 비밀번호 역시 파일 또는 평문 형태로 노출될 수 있는 구조다.

이 수업에서는 이 구조를 **“실운영 관점에서 납득 가능한 최소한의 방어선”** 수준으로 끌어올리는 것이 목표입니다.

---

## 변경 작업 1 – auth-profiles.json에서 Gemini API 키 제거

1. `auth-profiles.json` 파일을 열어, Gemini API 키가 들어 있는 필드를 찾습니다.
2. 해당 키 값(실제 문자열) 부분을 제거하거나, 더 이상 사용하지 않도록 구조를 정리합니다.
3. 이 파일은 더 이상 **API 키 저장소가 아니라, 프로필/구성 정보 위주**로만 사용되게 됩니다.

> 결과적으로, 저장소에 올라가거나 화면 공유를 하더라도  
> `auth-profiles.json` 안에서 **실제 키 값이 노출되지 않도록** 만드는 것이 목적입니다.

---

## 변경 작업 2 – 호스트 환경변수에서 Gemini API 키 읽기

1. 우분투(호스트)에서 환경변수를 등록합니다. (예: `.bashrc`, `.zshrc`, 시스템 환경설정 등)
   - 예시: `export GEMINI_API_KEY="여기에_실제_키_값"`
2. `docker-compose.yml`의 `environment:` 섹션을 수정해,  
   컨테이너가 이 환경변수를 그대로 받아 쓰도록 설정합니다.
   - 예시:
     ```yaml
     environment:
       - GEMINI_API_KEY=${GEMINI_API_KEY}
     ```
3. 이렇게 하면:
   - 실제 키 값은 **호스트 OS 환경변수에만 존재**하고,
   - `docker-compose.yml` 자체에는 평문 키가 남지 않습니다.

---

## 변경 작업 3 – --allow-configured 옵션 제거

1. `docker-compose.yml` 안의 `command:` 항목을 찾습니다.
2. 기존에 포함되어 있던 `--allow-configured` 옵션을 제거합니다.
3. 이 변경을 통해:
   - 컨테이너가 **임의 설정 파일이나 추가 경로를 과도하게 허용하는 위험**을 줄이고,
   - 미리 의도한 범위 내에서만 설정이 적용되도록 제한합니다.

> 이 옵션은 개발 단계에서는 편리할 수 있지만,  
> 교육/실습에서 **보안의 중요성을 강조하기 위해 제거**하는 방향으로 안내합니다.

---

## 변경 작업 4 – 토큰 비밀번호 환경변수화

1. 토큰 비밀번호를 지금 어디에 어떻게 저장하고 있는지 먼저 확인합니다.
   - 예: 설정 파일(`openclaw.json`)에 직접 문자열로 포함되어 있을 수 있습니다.
2. 이 값을 **호스트 환경변수**로 옮깁니다.
   - 예시: `export OPENCLAW_GATEWAY_TOKEN="여기에_토큰_비밀번호"`
3. `docker-compose.yml`의 `environment:`에 추가합니다.
   - 예시:
     ```yaml
     environment:
       - OPENCLAW_GATEWAY_TOKEN=${OPENCLAW_GATEWAY_TOKEN}
     ```
4. 애플리케이션이 이 환경변수를 참조하도록 설정되어 있는지 확인합니다.

---

## 이 수업 이후 기대되는 상태

이 수업을 마치면, 이상적인 상태는 다음과 같습니다.

- Git 저장소 어디에도 **Gemini API 키, 토큰 비밀번호 같은 민감 정보가 평문으로 남아있지 않습니다.**
- 실제 비밀 값은:
  - 우분투(호스트) 환경변수에만 존재하고,
  - 컨테이너에는 환경변수 형태로만 전달됩니다.
- `docker-compose.yml`의 `command:`에서 불필요한 `--allow-configured` 옵션이 제거되어,  
  **“딱 필요한 만큼만 허용된”** 보다 보수적인 실행 환경이 됩니다.

---

## 정리

이 수업은 에이전트를 더 똑똑하게 만드는 것이 아니라,  
**“이미 잘 돌아가는 내 비서를 어떻게 지켜줄 것인가”**에 초점을 맞춥니다.

- 키와 비밀번호를 **파일이 아닌 환경변수로 옮기고**,  
- 과한 옵션을 제거해 **공격 표면을 줄이는 연습**을 통해,  
실제 현업에서 요구되는 최소 수준의 보안 감각을 자연스럽게 익히는 것이 목표입니다.
